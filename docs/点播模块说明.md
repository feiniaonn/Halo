# 点播模块说明

## 模块定位

点播模块负责导入和管理 TVBox 类订阅源，完成站点浏览、搜索、详情、播放地址解析和播放器联动。它是 Halo 里最复杂的媒体链路，前端负责工作台和状态机，Rust 后端负责配置解析、网络兼容、本地服务与播放辅助。

## 当前能力

- 导入单仓、多仓和远程订阅源，并对 TVBox 类配置做规范化处理。
- 浏览站点首页、分类、搜索结果与详情页，统一转换成桌面端可用的数据结构。
- 支持 CMS 直连源与 Spider/Jar/JS 源两条链路，并对本地扩展、远程扩展和内置兜底做兼容。
- 提供 `Spider daemon + local service + HLS relay + native mpv` 的组合播放能力。
- 支持解析链路诊断、播放阶段日志、请求取消和单飞控制，减少切源、切集时的拖尾和串结果问题。

## 技术结构

### 前端

- `src/pages/MediaPage.tsx`：点播主页面入口。
- `src/modules/media/hooks/useVodSourceController.ts`：点播工作台核心控制器，负责源加载、站点切换、搜索、详情和播放入口。
- `src/modules/media/services/tvboxConfig.ts`：TVBox 配置标准化、站点能力推断和解析入口。
- `src/modules/media/services/vodDetail.ts`：详情页数据装配。
- `src/modules/media/services/vodPlayback.ts`：播放地址解析主链路，现已附带结构化 `diagnostics`。
- `src/modules/media/services/vodPlaybackLogging.ts`：把诊断步骤格式化到日志，便于复现与排障。
- `src/modules/media/services/vodNativePlayer.ts`：原生播放器桥接层。
- `src/components/VodPlayer.tsx`：播放窗口，负责内核选择、relay、探测与失败降级。

### 后端

- `src-tauri/src/media_cmds/media_cmds_tvbox.rs`：TVBox 配置抓取、解包、嵌套页提取与解密代理兜底。
- `src-tauri/src/spider_cmds.rs`：Spider 命令入口、资源定位与任务取消命令。
- `src-tauri/src/spider_cmds_exec.rs`：Spider 调度层，优先走常驻 JVM daemon。
- `src-tauri/src/spider_daemon.rs`：Rust 侧 daemon 宿主，负责 JSON-RPC 通信、取消和生命周期管理。
- `src-tauri/src/spider_local_service.rs`：本地 `/bf`、`/proxy`、`/parse`、`/transport` 等接口实现。
- `src-tauri/src/vod_hls_relay.rs`：HLS manifest、segment 和附属资源 relay。
- `src-tauri/src/native_player.rs`：原生 mpv 子窗口嵌入与控制。

## 运行链路

1. 前端导入或选择订阅源。
2. `tvboxConfig` 与 Rust `media_cmds_tvbox` 协同把原始配置解析成桌面端站点模型。
3. 用户浏览、搜索或打开详情时，站点请求进入 CMS 直连路径或 Spider 路径。
4. Spider 路径优先进入常驻 JVM daemon；必要时再回退单次进程。
5. 播放阶段由 `vodPlayback` 解析最终地址，并根据规则选择直连、代理、relay 或原生 mpv。
6. 若链路出错，诊断信息会附着在结果或异常对象上，供 `VodPlayer` 日志直接输出。

## 关键资源

- `src-tauri/TVBox/`：本地 TVBox 运行时模板与静态资源，供本地 `/parse` 页面及部分 JS/扩展链路复用。
- `src-tauri/resources/jar/`：JVM bridge、兼容 jar 与 fallback spider 资源。
- `src-tauri/resources/java/`：桌面 Spider 所需的 Java runtime。

## 当前边界

- 点播链路仍高度依赖外部源站、解析站和远程 Spider 质量，后端只能增强兼容，不能保证外部资源永远可用。
- 常驻 daemon 已解决频繁冷启动和切源拖尾问题，但不同 Spider 的行为差异依旧较大，真正的兼容性仍要靠诊断数据持续收敛。
- 某些播放失败属于外部流媒体自身问题，例如损坏的 manifest、超时的解析站、被风控的中转地址，这类问题无法单靠桌面端完全消除。

## 适合继续演进的方向

- 继续收紧 `Spider -> relay -> 播放器` 三段式诊断，优先解决真实失败样本。
- 为站点能力和播放策略做更多缓存，减少重复解析与重复探测。
- 把工作台里高频错误做成可视化状态，而不是只留在日志中。
