# Halo 仓库清理执行清单

更新日期：2026-03-18

这份清单基于当前仓库实际状态整理，目标是把“可公开保留的内容”和“应移出 Git 或仅留本地的内容”分开，后续可以直接按这里执行。

## 0. 执行前提

- 当前工作区不是干净状态，除了清理对象以外，还存在点播、播放器、bridge 等功能改动。
- 后续执行清理时，不要把功能改动和清理改动混在一个提交里。
- 建议顺序：
  1. 先确认当前功能改动是否需要单独提交。
  2. 再开一个专门的清理提交处理本文列出的删除项。

## 4. 需要拆分处理，不能整目录误删

### 4.1 `一键更新测试/`

这个目录不能按“只有一个 Linux 模板文件”处理。当前它实际包含：

- 发布脚本：`01_自动发布.ps1`、`02_仅推送Linux服务.ps1`、`03_仅构建.ps1`、`_publish_common.ps1`
- 配置和模板：`deploy.config.json`、`halo-update.service`、`halo-update.service.template`、`README.txt`、`更新公告_模板.txt`
- Linux 侧辅助脚本：`Linux_*.sh`
- 本地/敏感/一次性文件：`.publish-secrets.json`、`密钥.txt`、`更新公告.txt`

执行建议：

- 保留：发布脚本、服务模板、README、模板公告。
- 改为仅本地保留：`.publish-secrets.json`、`密钥.txt`、`更新公告.txt`
- 处理方式：优先用 `git rm --cached` 把本地/敏感文件移出版本控制，同时补 `.gitignore`，不要直接误删整个目录。

### 4.2 `.spec-workflow/`

当前被跟踪的是模板文件，不是临时日志目录。

执行建议：

- 如果后续还要保留规范化需求/设计模板，继续保留。
- 如果已经不再使用这套工作流，再单独作为一个目录清理。

结论：它不是高优先级垃圾目录，不建议和临时产物一起顺手删。

## 7. 可直接照着执行的命令草案

下面命令只对应“第一批低风险清理”，执行前仍建议先确认当前功能改动是否要单独提交：

```powershell
git add -u build-out.txt final_lint.txt final_lint_utf8.txt home_lint.txt home_lint_utf8.txt lint_output.txt lint_output_final.txt tsc-out.txt tsc-out2.txt

git rm capture_lint.mjs clean_lint.mjs fix_lint.mjs fix_lint_2.mjs fix_lint_3.mjs fix_lint_4.mjs fix_warnings.mjs

git rm -r spider-diagnostics tools/decompiled-bridge com META-INF src-tauri/spider-bridge/bridge-unpack src-tauri/spider-bridge/patch-classes src-tauri/spider-bridge/profile-classes

git rm src-tauri/spider-bridge/sources-manual.txt
```

`一键更新测试/` 和 `.spec-workflow/` 不要直接套这组命令，先拆分再处理。
