# 点播解析操作详细文档（单仓点播权威版）

## 1. 范围与目标
- 当前只做单仓点播，不做多仓并行浏览。
- 根配置如果包含 `urls`，页面只允许选择一个活动子仓继续浏览，仍然属于“单仓模式”。
- 点播主链路目标固定为：
  - 载入源
  - 识别分仓/站点
  - 首页或分类或搜索
  - 详情
  - 线路/剧集
  - 播放
- 当前实现同时覆盖两类站点：
  - CMS 型：常见 `type 0/1`
  - Spider 型：常见 `type 3/4` 或 `api` 为 `csp_*`

## 2. 当前落地状态
- 已支持本地解析优先、第三方解密服务兜底。
- 已支持 HTML 包装页、候选链接递归探测、base64 负载识别。
- 已支持 CMS 与 Spider 共用详情弹窗。
- 已支持播放时统一走“先拿剧集 URL，再判断是否需要解析”的流程。
- 已支持站点 `playUrl` 优先于全局 `parses`。
- 已支持解析失败后仅在明确需要时切隐藏 WebView 浏览器解析。
- 已支持远程 `ext` 先拉取并缓存，再把最终文本传给 Spider 命令。

## 3. 源解析流程
1. 输入源地址。
2. 后端先尝试直接读取 JSON。
3. 如果返回的是注释前缀、包装对象、字符串包 JSON，则做宽松解包。
4. 如果返回的是 HTML 页面，则递归提取候选链接继续探测。
5. 如果返回的是疑似 base64 负载，则先解码再判断是否是 TVBox 配置。
6. 如果仍无法得到有效配置，再尝试已知第三方解密代理兜底。
7. 如果根配置包含 `urls`，前端只展示一个活动子仓，用户切换后重新加载该子仓配置。

## 4. 字段表（当前实现已识别）

### 4.1 根配置字段
- `spider`
  - Spider JAR 地址。可能带 `;md5;...` 尾部信息。
  - 站点级没有单独 `jar` 时，默认使用这里的 Spider。
- `sites`
  - 站点数组。
- `urls`
  - 多仓包装。当前只取一个活动子仓。
- `parses`
  - 全局解析器数组。播放时作为站点 `playUrl` 的后备解析链。
- `logo`
  - 源图标地址，当前保存在归一化配置中。
- `wallpaper`
  - 源背景地址，当前保存在归一化配置中。

### 4.2 站点字段
- `key`
  - 站点唯一标识。
- `name`
  - 站点展示名称。
- `type`
  - 站点类型。常见：
    - `0/1`：CMS 型
    - `3/4`：Spider 型
- `api`
  - CMS 型通常是接口地址。
  - Spider 型通常是 `csp_*` 类名或等价标识。
- `searchable`
  - 是否允许搜索。
- `quickSearch`
  - 是否允许快速搜索。
- `filterable`
  - 是否支持分类过滤。
- `jar`
  - 站点级 Spider/JAR 覆盖地址。
- `ext`
  - Spider 扩展参数。
  - 当前归一化后分为：
    - 纯文本
    - JSON 对象
    - JSON 数组
    - 远程 URL
  - 远程 URL 会在运行时先拉取并缓存，再传给 Spider。
- `playUrl`
  - 站点级解析地址。
  - 播放时优先于全局 `parses`。
- `click`
  - 浏览器解析时的可选点击提示。
  - 当前仅在隐藏 WebView 解析分支使用。
- `playerType`
  - 站点播放器类型标识，当前已进入播放器启动载荷。
- `categories`
  - 预设分类数组。无首页分类返回时可直接作为分类入口。

### 4.3 解析器字段 `parses[]`
- `name`
  - 解析器展示名。
- `type`
  - 当前只支持 `0/1` 进入自动解析链。
- `url`
  - 解析前缀地址。
- `ext.header`
  - 解析器附带请求头。
- `ext.flag`
  - 仅当线路名或页面地址匹配 `flag` 时参与候选。

## 5. 能力模型
每个站点会被归一化成统一能力，不再直接依赖原始 JSON：
- `sourceKind`
  - `cms` 或 `spider`
- `canHome`
  - 是否可以拉首页/推荐
- `canCategory`
  - 是否可以拉分类
- `canSearch`
  - 是否可以搜索
- `searchOnly`
  - 仅搜索型站点
- `displayOnly`
  - 仅展示型站点
- `supportsDetail`
  - 是否允许进入详情弹窗
- `supportsPlay`
  - 是否具备完整播放闭环
- `mayNeedParse`
  - 是否可能进入解析器链路
- `supportsBrowserParse`
  - 是否允许浏览器解析兜底
- `hasRemoteExt`
  - 是否含远程 `ext`
- `hasPlayUrl`
  - 是否含站点级 `playUrl`
- `hasPresetCategories`
  - 是否含预设分类

## 6. 首页、分类、搜索流程

### 6.1 首页
- CMS：走 `fetch_vod_home`
- Spider：先解析运行态 `ext`，再走 `spider_home`
- 如果首页没有返回分类，但站点存在 `categories`，前端直接使用预设分类

### 6.2 分类
- CMS：走 `fetch_vod_category`
- Spider：先解析运行态 `ext`，再走 `spider_category`

### 6.3 搜索
- CMS：走 `fetch_vod_search`
- Spider：先解析运行态 `ext`，再走 `spider_search`
- 搜索型站点允许跳过首页/分类，直接从搜索结果进入详情

## 7. 详情流程
- 详情弹窗已经改为源无关：
  - CMS：走 `fetch_vod_detail`
  - Spider：走 `spider_detail`
- CMS 详情命令优先使用：
  - `ac=detail&ids=`
- 如果上面失败，再回退：
  - `ac=videolist&ids=`
- 返回结构统一对齐为 `VodDetail` + `VodRoute[]`
- 剧集 URL 中如果是 `msearch:`，前端视为“搜索入口”，不当作直链播放

## 8. 播放与解析流程
1. 用户选择线路和剧集。
2. 先拿当前剧集 URL。
3. 如果是 Spider 站点：
   - 先走 `spider_player`
   - 如果 Spider 直接给出真实媒体 URL，则直接播放
   - 如果返回 `parse=1` / `jx=1`，或没有给出可播放 URL，则进入解析链
4. 如果是 CMS 站点：
   - 如果剧集 URL 本身看起来是媒体直链，则直接播放
   - 否则进入解析链
5. 解析链优先级固定为：
   - 站点 `playUrl`
   - 全局 `parses`
6. 解析器筛选规则：
   - 仅使用 `type 0/1`
   - 若解析器定义了 `ext.flag`，则需匹配线路名或页面 URL
7. 解析执行顺序：
   - 先走 `resolve_jiexi`
   - 若明确返回 `jiexi_needs_browser`，再走 `resolve_jiexi_webview`
8. `click` 只在浏览器解析分支中参与，不影响直链与 JSON 解析分支。

## 9. 错误与回退策略
- 源解析失败
  - 优先提示本地解析失败
  - 如果连兜底也失败，再提示未获得有效配置
- Dex Spider 兼容失败
  - 如果 Spider JAR 是 `classes.dex` 且不含 `.class`，桌面端会先自动做一次 `dex -> desktop jar` 转换
  - 如果转换失败，提示：`Spider dex 转桌面 jar 失败`
  - 如果转换成功但目标站点方法仍是 `native`，提示：`Spider 依赖 Android JNI/ARM 原生库，桌面端暂时无法直接执行`
  - 对这类失败，不再偷偷回退到内置 `AppYsV2`，避免出现“固定默认分类 + 空列表”的假可用状态
- Spider 执行路由
  - `JvmJar`：直接走桌面桥接
  - `DexOnly`：先生成 `desktop.jar`，再走桌面桥接
  - `DexNative`：直接标记为 `需要 Android Worker`
  - 一旦已经命中真实站点类并执行过方法，不再回退到内置 `halo_spider.jar`
- 桌面 Spider 基础运行库
  - 当前固定随应用分发：
    - `json.jar`
    - `okhttp.jar`
    - `okio.jar`
    - `gson.jar`
    - `commons-lang3.jar`
  - 目标是把“运行中撞缺类”改成显式依赖集
- 统一错误分类
  - `FetchError`
    - Spider 下载、缓存、路径、JAR 读取失败
  - `TransformError`
    - dex 转换失败或转换超时
  - `MissingDependency`
    - 典型例子：`NoClassDefFoundError: com/google/gson/reflect/TypeToken`
  - `NeedsContextShim`
    - 站点声明了 `init(Context, ...)`，需要先经过桌面 `Context shim`
  - `NeedsCompatPack`
    - 需要补挂兼容包或兼容包准备失败
  - `NeedsLocalHelper`
    - 需要桌面本地 helper 服务参与
  - `NativeMethodBlocked`
    - 命中 `native` / JNI / `.so` 路径，但当前桌面兼容层尚未接住
  - `ClassSelectionError`
    - 站点 key / hint 未命中真实类
  - `InitError`
    - Spider 初始化阶段空指针、非法状态等
  - `SiteRuntimeError`
    - 业务执行阶段异常，但不属于缺依赖 / JNI
  - `ResponseShapeError`
    - Spider 返回结构异常，或桥接未返回有效结果
  - `Timeout`
    - Spider 执行超时
- 详情失败
  - 空响应：`源站返回空详情`
  - 结构不对：`未获取到影视详情`
- 播放失败
  - 图片流：提示返回的不是视频流
  - 无直链且无解析器：提示当前剧集需要解析，但没有可用解析器
  - 解析器全失败：提示解析链路未返回可播地址
  - 浏览器解析超时：保留 `jiexi_webview_timeout` 作为明确错误

### 9.1 前端运行态显示规则
- 点播页当前接口卡片会展示：
  - 当前执行器：`desktop-direct / desktop-compat-pack / desktop-helper`
  - 当前状态：`桌面直跑 / 兼容包路径 / 本地 Helper / 缺依赖 / 站点异常 / 临时禁用`
  - 最近一次错误分类
  - 最近一次失败消息
  - 已准备的 `preparedJarPath`
  - 已识别的 `nativeLibs`
  - 已命中的 `requiredCompatPacks`
  - 已命中的 `requiredHelperPorts`
- 接口切换弹窗不会再平铺占用主内容区，只保留当前接口摘要和状态。
- 同类失败连续达到阈值后，接口会进入 `临时禁用`，避免页面每次进入都自动重试同一批错误。

### 9.2 桌面兼容层主线（2026-03-07）
- 当前主方案已经切到 `Windows 主产品 + 桌面兼容层 + 应用自带本地 helper`：
  - `DesktopDirect`
    - 当前 bridge 直接执行
  - `DesktopCompatPack`
    - 自动补挂兼容包，必要时把 compat jar 也走 `dex -> desktop.jar`
  - `DesktopHelper`
    - 自动拉起本地 helper，再继续执行站点逻辑
- 当前桌面基础运行库固定随应用分发：
  - `json.jar`
  - `okhttp.jar`
  - `okio.jar`
  - `gson.jar`
  - `commons-lang3.jar`
  - `commons-codec.jar`
  - `commons-io.jar`
  - `guava.jar`
  - `jdom2.jar`
  - `jsoup.jar`
  - `juniversalchardet.jar`
  - `kotlin-stdlib.jar`
  - `fastjson.jar`
  - `rhino.jar`
  - `slf4j-api.jar`
  - `slf4j-simple.jar`
- 当前桌面兼容包目录：`src-tauri/resources/jar/compat`
  - `legacy-core`
    - `base.jar`
    - `spider.jar`
  - `legacy-jsapi`
    - `custom_jsapi.jar`
    - `remote_spider.jar`
  - `legacy-custom-spider`
    - `custom_spider.jar`
- 兼容包不会全量污染 classpath：
  - 前端画像后返回 `requiredCompatPacks`
  - 后端只按当前站点命中的 pack 补挂
- `android-worker/` 继续保留，但现在只作为实验归档，不再是桌面主流程的一部分。

### 9.3 本地 helper 协议矩阵
- helper 形态：
  - `compat_helper.jar`
  - 由桌面端统一启动、探活和回收
  - 当前最小能力：`/health`、`/trace/last`、`/trace/reset`、通用代理转发
- 已知协议入口：
  - `127.0.0.1:9966`
    - 来源：`commonConfig.goServerUrl`
    - 当前策略：由 helper 兜底代理并记录目标 URL / 请求形状
  - `127.0.0.1:1072`
    - 来源：`commonConfig.proxy`
    - 当前策略：由 helper 兜底代理并记录目标 URL / 请求形状
  - `127.0.0.1:9999`
    - 来源：`Init.class` 固化地址
    - 当前策略：由 helper 兜底代理并记录目标 URL / 请求形状
- 当前 helper 的职责不是“假返回分类”，而是：
  - 先提供真实本地端口
  - 先把请求录出来
  - 先尝试最小转发
  - 后续再针对 Track A 样本补具体协议语义

### 9.4 当前真实样本结论（2026-03-07）
- `TgYunDouBanPan`
  - `homeContent()` 为 `native`
  - 当前会被明确归到 `NativeMethodBlocked`
  - 不再回退假分类，也不再伪装成 Android Worker 主线
- `GuaZi`
  - `gson` 缺依赖已补齐，不再停在 `TypeToken`
  - 当前会先走：画像 -> compat pack -> helper probe
  - 若仍失败，明确落到 `InitError / SiteRuntimeError / NeedsLocalHelper`
- `TTian`
  - 当前会先走：画像 -> Context shim -> compat pack -> helper probe
  - 不再因为声明了 `Context init` 就被首轮提前判死
- `commonConfig` 额外发现
  - `peizhi.json` 内含本地依赖端口：
    - `goServerUrl: http://127.0.0.1:9966`
    - `proxy: http://127.0.0.1:1072`
  - `Init.class` 还固化了 `http://127.0.0.1:9999`
  - 这说明部分 Spider 默认假设存在 companion/本地服务，而不是单靠 JAR 自给自足
