## 最重要的结论

1. `iyouhun` 当前是相对稳定的。
   `dc-xs` 根地址会先解析成 `6` 个分仓，首仓 `movie.json` 的 `AppRJ` 在 debug 和 release 都能稳定跑到“有分类、有影视列表”。

2. `iyouhun` 不是所有 Spider 都稳定。
   `Douban` 已经能通，`Hxq` 目前只到 `profile/home`，还没产出分类和列表；`BiliYS` 仍然卡在 `Spider.safeDns()` 的签名不兼容。

3. `feimao` 根入口本身就是正式版差异的主要来源。
   debug 这轮稳定解析到了远端 `XC.json`，得到 `81` 个站点；release 这轮批量脚本里则出现“根抓取失败，回退到内置 `12` 站点快照”的情况。也就是说，正式版里同一个站点序号并不稳定。

4. `feimao` 的失败主要分成 4 类。
   `NeedsLocalHelper`：例如 `GuaZi`、`ConfigCenter`，依赖本地 helper 端口 `1072/9966`。
   `SiteRuntimeError`：例如 `DouBan`、`TTian`、`CzzyAmns`，是 Spider 站点自身空指针或内部状态未初始化。
   `Timeout`：例如 `Jpys`。
   `ResponseShapeError`：例如 `YGP`。

5. `bridge.jar` 仍然是一个环境差异点。
   runtime 的 [bridge.jar](/d:/Development/Projects/Halo/src-tauri/resources/jar/bridge.jar) 不包含 `SpiderProfileRunner`，诊断时实际依赖的是 [bridge.new.jar](/d:/Development/Projects/Halo/src-tauri/spider-bridge/bridge.new.jar)。这会持续放大 debug / release 的链路差异。

## 接下来最值得做的事

- 先把 `feimao` 根入口的“远端解析成功 / bundled fallback”分叉稳定下来，否则站点序号永远会漂。
- 给 `BiliYS` 单独补 `safeDns()` 兼容链路，否则它不会进入“有分类、有影视”的状态。
- 对 `GuaZi / ConfigCenter` 判断本地 helper 是否真的属于正式版依赖；如果不是，就应该在 UI 层直接标记不可用，而不是让它们进入点播主链路。
